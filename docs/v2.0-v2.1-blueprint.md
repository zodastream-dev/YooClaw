## 二、V2.0：_valueScore 打分机制

### 2.1 核心设计原则

DeepSeek 打分容易出现的三个问题及对策：

| 问题 | 对策 |
|---|---|
| **虚高**（所有条目打 85+） | 明确"60分是合格情报的锚点"，用 Few-Shot 给出各级别典型示例 |
| **趋中**（大量集中在 40-60） | 要求分数必须呈"金字塔分布"：10%在 80+、30%在 60-80、60%在 60 以下 |
| **不一致**（同一条目不同次打分差异大） | 绑定具体判断维度，每维度有明确的是/否条件，减少主观空间 |

### 2.2 Prompt 修改方案

在 `callIntel()` 中的用户提示词（`intel-pipeline.ts` 第 370-389 行）追加以下内容：

**追加到 JSON Schema 中**（在 `_intent` 字段后新增）：

```
"_valueScore": <正整数 0-100，商业价值评分>
```

**追加到 Prompt 要求中**（在现有第 10 条后新增）：

```
11. _valueScore 商业价值判分标准（0-100）：
   【90-100 战略级】影响投资决策或战略方向调整的信息：
     - 官方财报发布、重大并购、核心高管变更、监管政策突变
     - 行业龙头市场份额大幅变化（>5%）、颠覆性技术突破

     - 竞品新品发布、关键供应链变动、大客户中标/流失
     - 技术标准更新、重要合作伙伴动态
     - 判法：是否需要某个业务部门做出具体应对？是→75+
   【60-74 关注级】值得了解的行业动态：
     - 行业趋势报告、市场数据更新、一般性产品迭代
```

## 三、V2.1：晨报内参聚合 Prompt

### 3.1 核心挑战

将 3-5 条独立 JSON 情报融合成一篇文章，而非摘要堆砌。关键技巧：

- **叙事线**：找到多条情报之间的逻辑关联（同一事件链？同一趋势？互相矛盾？）
- **优先级**：最高分的情报作为"今日头条"，其余作为"关联阅读"
- **行动导向**：每段都以"这意味着……"或"建议关注……"收尾

### 3.2 Prompt 模板

```typescript
function makeBriefingPrompt(
  portalName: string,
  highValueIntel: Array<{ title: string; summary: string; source: string; _valueScore: number }>
): { system: string; user: string } {
  const today = new Date().toLocaleDateString('zh-CN', {
    year: 'numeric', month: 'long', day: 'numeric', weekday: 'long'
  });
  const intelText = highValueIntel.map((item, i) =>
    `${i + 1}. [价值分${item._valueScore}] ${item.title}
   摘要：${item.summary}
   来源：${item.source}`
  ).join('\n\n');

  const system = `你是顶级战略顾问，为「${portalName}」的高管撰写每日晨报内参。
你的文字风格：像资深幕僚而非新闻聚合器——精准、冷静、有判断力。

写作原则：
1. 不是摘要拼凑，而是"把几条新闻讲成一个故事"
2. 先给结论（3句话内讲完今日最有价值的信息），再展开细节
3. 每条关键信息后附"研判"——这不是描述，是你的专业判断
4. 用 Markdown 格式：**加粗关键数据**、用 > 引用原文关键信息
5. 如果需要决策建议，用 📌 标记
6. 总长度控制在 300 字左右（不含标题）

禁止：
- 不要写"根据搜索结果""据悉""据报道"等冗余措辞
- 不要复读摘要，要提炼核心
- 不要用模糊词汇（"可能""或将""有望"），有把握就说，没把握就写"待确认"

输出格式：
## 📊 {今日日期} 晨报
（正文）`;

  const user = `今天是${today}。

以下是过去24小时价值最高的情报（共${highValueIntel.length}条）。请基于这些内容生成高管晨报内参。

情报列表：
${intelText}

请直接输出晨报，不要任何前缀说明。`;

  return { system, user };
}
```

### 3.3 调用方式

```typescript
async function generateBriefing(
  portalName: string,
  highValueIntel: any[],
  model: string = 'deepseek-v4-flash'
): Promise<string> {
  const { system, user } = makeBriefingPrompt(portalName, highValueIntel);

  const resp = await fetch('https://api.deepseek.com/chat/completions', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'Authorization': 'Bearer ' + (process.env.DEEPSEEK_API_KEY || ''),
    },
    body: JSON.stringify({
      model,
      max_tokens: 800,
      temperature: 0.4,  // 偏低，保证连贯性和稳定性
      messages: [
        { role: 'system', content: system },
        { role: 'user', content: user },
      ],
    }),
    signal: AbortSignal.timeout(30000),
  });

  if (!resp.ok) throw new Error('Briefing generation failed: HTTP ' + resp.status);
  const data = await resp.json();
  return data.choices[0].message.content.trim();
}
```

## 四、V2.1：微信推送代码

### 4.1 技术选型

两个主流方案对比：

| 稳定性 | 偶尔限流 | 较好 |

### 4.2 PushPlus 集成代码

```typescript
/**
 * Send a Markdown message via PushPlus.
 * @param token  PushPlus token (from https://www.pushplus.plus)
 * @param title  Message title (shown in push notification)
 * @param content Markdown content
 * @param template  'markdown' for Markdown rendering
 */
async function pushWechatMessage(
  token: string,
  title: string,
  content: string,
  template: 'html' | 'markdown' = 'markdown'
): Promise<boolean> {
  try {
    const resp = await fetch('https://www.pushplus.plus/send', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({
        token,
        title,
        content,
        template,      // 'markdown' = Markdown rendering
        channel: 'wechat',
      }),
      signal: AbortSignal.timeout(10000),
    });

    if (!resp.ok) {
      console.error('[BriefingPush] PushPlus HTTP error:', resp.status);
      return false;
    }

    const data = await resp.json();
    if (data.code === 200) {
      console.log('[BriefingPush] Sent successfully:', title);
      return true;
    } else {
      console.error('[BriefingPush] PushPlus API error:', data.code, data.msg);
      return false;
    }
  } catch (e: any) {
    console.error('[BriefingPush] Failed:', e.message);
    return false;
  }
}
```

## 五、非阻塞架构调度方案

### 5.1 现状分析

当前 index.ts 中的定时任务（第 5838-5853 行）：

```
┌─────────────────────────────────────┐
│  进程启动                            │
│  ├── setTimeout 30s: warmAllPortalCaches()
│  ├── setInterval 5min: savePortalIntelCache()
│  └── setInterval 20min: warmAllPortalCaches()  ← 有锁（cacheWarmingActive）
│       └── 子：3AM nightly full refresh
└─────────────────────────────────────┘
```

**关键约束**：`warmAllPortalCaches()` 已有 `cacheWarmingActive` 互斥锁（第 3727 行），不会并发执行。但 DeepSeek 调用会阻塞整个进程（单线程 Node.js 事件循环）。

### 5.2 晨报任务设计（推荐方案：懒启动 + 独立锁）

```typescript
// ========== V2.1 Daily Briefing Scheduler ==========

/** Whether a briefing push is currently in progress */
let briefingActive = false;

/** Track the last date a briefing was sent for each portal (prevent duplicate) */
const lastBriefingDate = new Map<string, string>(); // slug → YYYY-MM-DD

/**
 * Execute the daily briefing pipeline for a single portal:
 * 1. Fetch fresh intel (from cache or live)
 * 2. Select top-N by _valueScore
 * 3. Generate briefing via DeepSeek
 * 4. Push via PushPlus/Server酱
 *
 * Designed to run independently — does NOT block CacheWarmer or main event loop.
 * Uses its own lock (briefingActive) — separate from cacheWarmingActive.
 */
async function runDailyBriefing(slug: string) {
  const today = new Date().toISOString().slice(0, 10);

  // Prevent duplicate run today
  if (lastBriefingDate.get(slug) === today) {
    console.log(`[Briefing] ${slug}: already sent today, skipping`);
    return;
  }

  if (briefingActive) {
    console.log(`[Briefing] Another briefing is in progress, skipping ${slug}`);
    return;
  }

  briefingActive = true;
  try {
    // Step 1: Get portal config + push token
    const site = await getPortalSite(slug);
    if (!site) { console.log(`[Briefing] ${slug}: portal not found`); return; }

    const pushToken = extractPushToken(site);  // from html_content or DB
    if (!pushToken) { console.log(`[Briefing] ${slug}: no pushToken configured`); return; }

    // Step 2: Get cached intel (reuse existing pipeline)
    const intelData = await getPortalIntel(slug);  // from portalIntelCache or fresh fetch
    if (!intelData || intelData.length === 0) {
      console.log(`[Briefing] ${slug}: no intel data`);
      return;
    }

    // Step 3: Select top-N by _valueScore
    const topN = intelData
      .filter((item: any) => (item._valueScore || 0) >= 60)  // 仅推送关注级以上
      .sort((a: any, b: any) => (b._valueScore || 0) - (a._valueScore || 0))
      .slice(0, 5);

    if (topN.length === 0) {
      console.log(`[Briefing] ${slug}: no high-value intel today (>60)`);
      return;
    }

    // Step 4: Generate briefing via DeepSeek (non-blocking: this is its own async flow)
    console.log(`[Briefing] ${slug}: generating briefing from ${topN.length} items...`);
    const portalName = site.title || slug;
    const briefing = await generateBriefing(portalName, topN);
    console.log(`[Briefing] ${slug}: briefing generated (${briefing.length} chars)`);

    // Step 5: Push to WeChat
    const success = await pushWechatMessage(
      pushToken,
      `${portalName} · 每日情报晨报 (${today})`,
      briefing,
      'markdown'
    );

    if (success) {
      lastBriefingDate.set(slug, today);
      console.log(`[Briefing] ${slug}: pushed successfully`);
    }
  } catch (e: any) {
    console.error(`[Briefing] ${slug}: failed —`, e.message);
  } finally {
    briefingActive = false;
  }
}

/**
 * Main daily briefing scheduler — runs at 8:00 AM every day.
 * Iterates ALL portals that have pushToken configured.
 */
async function runAllDailyBriefings() {
  console.log(`[Briefing] 8:00 AM — starting daily briefing round`);
  try {
    const allSites = await getAllPortalSites();
    const pushedCount = 0;

    // Process portals sequentially to avoid overwhelming APIs
    for (const site of allSites) {
      await runDailyBriefing(site.slug);
      // Short delay between portals (avoid rate limiting on DeepSeek/PushPlus)
      await new Promise(r => setTimeout(r, 2000));
    }

    console.log(`[Briefing] Round complete.`);
  } catch (e: any) {
    console.error(`[Briefing] Scheduler error:`, e.message);
  }
}
```

### 5.3 挂载到 index.ts

在 `index.ts` 的启动块中（第 5853 行附近）添加：

```typescript
// ===== V2.1 Daily Briefing: 8:00 AM =====
// Runs independently from CacheWarmer — separate lock, separate timing.
// Coexists peacefully because:
//   1. CACHE_WARMER runs every 20min, BRIEFING runs once at 8AM — natural time separation
//   2. Each has its OWN lock (cacheWarmingActive vs briefingActive) — no deadlock risk
//   3. Briefing READS from cache (getPortalIntel) — if cache is being written,
//      it either gets the previous cached version or waits for warmer to finish
setInterval(() => {
  const now = new Date();
  if (now.getHours() === 8 && now.getMinutes() < 20) {  // 20-min window for retry
    const today = now.toISOString().slice(0, 10);
    // Prevent duplicate: only run if we haven't started a round today
    runAllDailyBriefings().catch(e =>
      console.error('[Briefing] Scheduler crash:', e.message)
    );
  }
}, 10 * 60 * 1000);  // Check every 10 minutes
```

### 5.4 为什么不阻塞

```
时间线示例：

08:00  ──── Briefing Round 开始（独立锁 briefingActive=true）
           ├── Portal A: getIntel(cache hit, 0ms) → generateBriefing(15s) → push(0.5s)
           └── Portal B: getIntel(cache hit, 0ms) → generateBriefing(12s) → push(0.5s)
08:01  ──── Briefing Round 仍在进行中
08:20  ──── CacheWarmer 触发
           cacheWarmingActive 检查 → false（空闲） → 开始预热（独立锁 cacheWarmingActive=true）
           → Briefing 和 Warmer 各自持有自己的锁，互不冲突
08:30  ──── Briefing Round 完成（briefingActive=false）
```

**关键设计原则**：
1. **时间分离**：Briefing 每天只在 8:00 触发一次，Warmer 每 20 分钟触发——绝大多数时间不重叠
2. **独立锁**：`briefingActive` 和 `cacheWarmingActive` 各自独立
3. **只读共享**：Briefing 只从缓存读取（`getPortalIntel`），不修改缓存；Warmer 写入自己的 KV cache
4. **事件循环不阻塞**：所有 DeepSeek 调用都是 `async/await` + `fetch`，I/O 等待期间事件循环可处理其他请求

## 七、风险与注意事项

1. **打分稳定性**：建议上线前跑人工抽检——取 3 个不同类型门户各 1 次完整情报流，人工标注 50 条，与 DeepSeek 打分做 Pearson 相关性校验。目标 >0.7。

2. **晨报质量**：如果当天只有 1-2 条高价值情报，300 字要求可能导致 DeepSeek "注水"。解决方案：`topN.length < 3` 时自动跳过（不发晨报），避免推送低质量内容。

3. **PushPlus 免费额度**：200 条/天，按 50 个门户、每天发 1 次计算完全够用。但要注意 PushPlus 有 message 长度限制（约 2000 字符），晨报 300 字远低于此。

4. **8:00 准时性**：当前方案使用 `setInterval` 每 10 分钟检查一次小时数，精度约 ±5 分钟。如需精确到分钟级，可用 `node-cron` 替代，但增加一个依赖。建议先用 `setInterval` 方案（零依赖），观察效果后再决定。

5. **存量门户**：V2.0 需要批量 re-deploy 所有门户才能生效新排序逻辑；V2.1 的 pushToken 是可选的，没有配置的门户自动跳过。
